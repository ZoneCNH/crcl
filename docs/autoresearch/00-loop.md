# Autoresearch 迭代循环

## 结论

CRCL 文档版 autoresearch 不是自动扩写，而是固定预算的研究实验循环。

每次迭代只做一件事：提出一个可验证假设，用固定指标检查，保留能提高框架可用性的改动，丢弃没有稳定证据或无法复盘的改动。

## 方法映射

| Autoresearch 元素 | ML 版本           | CRCL 研究版本                    |
| ----------------- | ----------------- | -------------------------------- |
| 目标函数          | validation metric | 框架解释力和决策可执行性         |
| 最小实验          | 改一处训练配置    | 改一条研究假设、指标、阈值或模板 |
| 运行              | 训练 5 分钟       | 核验官方源、链上数据、财务口径   |
| 评估              | 指标提升或下降    | 结论是否更准、更快、更可复盘     |
| 保留              | commit 改动       | 写入对应 docs 文件               |
| 丢弃              | revert 改动       | 记录为 rejected 或 deferred      |

validation metric 是验证指标，白话就是每次实验都用同一把尺子比较结果。

## 目标函数

```text
CRCL research score =
  30% factual accuracy
  + 20% decision usefulness
  + 15% risk sensitivity
  + 15% valuation linkage
  + 10% source traceability
  + 10% maintenance simplicity
```

factual accuracy 是事实准确性，白话就是不能把过期数据、二级来源或推测当成结论。

## 单次迭代定义

一次迭代必须包含五项：

1. 假设：需要验证的研究判断。
2. 指标：用什么数据或文件判断。
3. 实验：查一处来源、调整一条阈值、补一个模板或删除一个弱判断。
4. 评估：是否提高目标函数。
5. 处置：keep、revise、defer、reject。

keep 是保留，白话就是写进正式框架。
defer 是延后，白话就是等财报、监管或链上数据再判断。

## 保留规则

满足任意两条即可保留：

1. 指标能被官方源或稳定数据源验证。
2. 能改变日、周、季、事件流程中的动作。
3. 能降低误判概率。
4. 能把叙事映射到财务、监管、竞争或估值。
5. 能让下一次复盘更快。

## 丢弃规则

满足任意一条就丢弃：

1. 只有观点，没有指标。
2. 只有新闻情绪，没有官方源。
3. 只能解释过去，不能指导下一次复盘。
4. 和现有框架重复，但没有增加阈值、来源或动作。
5. 需要不可获得的数据，且没有替代指标。

## Defer 条目复审机制

`01-iteration-log.md` 中标记为 `defer` 的条目不是永久搁置，而是等待触发条件再判断。

### 复审触发条件

| 触发 | 需复审的 defer 类型 |
| ---- | ------------------ |
| 季度财报发布 | defer 原因含"等财报"、"等披露"、"等电话会" |
| 监管最终规则发布 | defer 原因含"看最终规则"、"等监管" |
| 数据源稳定后 | defer 原因含"数据源需固定"、"需要 API 权限" |
| 框架自检（每季度） | 全部 defer 条目扫一遍，判断是否可以 keep 或 reject |

### 复审判断规则

| 复审结果 | 条件 | 动作 |
| -------- | ---- | ---- |
| 升级为 keep | 触发条件已满足，有稳定来源，改变现有框架 | 写入对应文件，日志标记为 keep |
| 维持 defer | 触发条件未满足，或数据仍不可获取 | 记录复审日期，更新 defer 理由 |
| 降级为 reject | 触发后发现假设不成立，或框架已有更好覆盖 | 日志标记为 rejected，不写入文件 |

### 快速分类（减少重复工作）

当前 140 条 defer 中，大部分属于以下模式，可按类批量处理：

| 模式 | 条目数（约） | 复审逻辑 |
| ---- | ------------ | -------- |
| "无稳定源则只做观察"（0251-0260 系列） | ~30 | 每季财报后扫一次，有稳定来源时 keep，否则维持 |
| "等电话会问答" | ~20 | 每次财报电话会后 24h 内复审 |
| "等最终规则" | ~15 | GENIUS Act OCC/Treasury 最终规则发布后复审 |
| "数据源需固定" | ~15 | 下次框架自检时决定是否找到替代来源 |
| "等财报拆分" | ~10 | 每季财报后检查 Other revenue 是否拆分 |
| 其他 | ~50 | 按季度自检时逐条看 |

复审时不需要全部 140 条逐一重写，只要把升级为 keep 的写入对应文件，并在日志里更新处置为 keep 即可。

## 目标函数版本控制

目标函数权重不是固定的，每次框架自检后可调整。每次调整必须记录原因和生效日期。

| 版本 | 生效日期 | 变更内容 | 变更原因 |
| ---- | -------- | -------- | -------- |
| v1.0 | 2026-05-08 | 初始权重（见下） | 框架建立基准 |

**v1.0 权重（当前版本）**：
```text
CRCL research score =
  30% factual accuracy
  + 20% decision usefulness
  + 15% risk sensitivity
  + 15% valuation linkage
  + 10% source traceability
  + 10% maintenance simplicity
```

**权重调整规则**：
- 单次调整幅度不超过 ±5ppts，禁止单次大幅重构
- 调整前必须说明"哪类错误最频繁"作为依据
- 调整后至少运行 10 次迭代再评估是否有效

**可能触发权重调整的信号**：

| 信号 | 建议调整方向 |
| ---- | ------------ |
| 频繁出现"结论正确但不知道该做什么"的迭代 | 提高 decision usefulness 权重 |
| 频繁出现"来源过期导致结论失效"的迭代 | 提高 factual accuracy 权重 |
| 风险漏判（事后发现 P0 预警信号早已出现） | 提高 risk sensitivity 权重 |
| 估值和业务结论脱节 | 提高 valuation linkage 权重 |
| 框架文件越来越难维护 | 提高 maintenance simplicity 权重 |

## 历史实验案例

### 正面案例（已 keep 的实验）

**案例 P-01：RLDC margin 38% 告警线**
- 假设：RLDC margin 低于某阈值时自动触发 Bear 复核，不依赖主观判断
- 实验：基于 FY2025 RLDC margin 42% 和 Bear 情景 38% 假设，设定自动告警线
- 评估：消除了"margin 下滑时是否需要复核"的歧义，decision usefulness +明显
- 处置：keep → 写入 `metrics/00-metric-dictionary.md` P0 告警条件
- 学习：单一量化阈值比"持续下滑"定性描述更容易触发动作

**案例 P-02：竞争评分 7 维度打分细则**
- 假设：竞争评分如果没有每档的具体数字标准，不同人打分会出现±20分差异
- 实验：为7个维度各设0/25/50/75/100分的具体条件（基于市占率/增速/监管等数字）
- 评估：消除了打分主观性，使两人评分差从±20降低到理论±10以内
- 处置：keep → 写入 `metrics/05-competition-scoring-rubric.md`
- 学习：定性描述必须配数字才能产生可重复的打分结果

**案例 P-03：跨文件触发链接**
- 假设：预警信号触发后，不同文件的动作未联动，导致触发了信号但没有后续动作
- 实验：在 `framework/00-research-map.md` 建立全局触发阈值索引，明确"触发A→执行B→更新C"链
- 评估：解决了"信号孤岛"问题，risk sensitivity 和 decision usefulness 均提升
- 处置：keep → 写入 `framework/00-research-map.md` 跨文件联动规则
- 学习：同一阈值若出现在多个文件中，必须有一个主文件作为权威源

### 负面案例（reject 或 defer 的实验）

**案例 N-01：实时价格波动触发仓位调整**
- 假设：CRCL 股价单日下跌 >5% 时触发仓位减仓检查
- 实验：在 playbook 中加入股价监控触发条件
- 评估：股价日波动属于市场噪声，与基本面框架定位不一致；且缺乏止损逻辑会导致频繁误触
- 处置：reject → 股价信号不进入 P0/P1 框架，基本面变化才触发动作
- 学习：触发条件必须与框架核心指标（RLDC/竞争/监管）直接相关，纯价格信号不适合基本面框架

**案例 N-02：Arc 链上数据实时监控**
- 假设：将 Arc 测试网日活交易量加入 P0 日监清单
- 实验：尝试从 Dune 查询 Arc 测试网交易量
- 评估：Arc 测试网数据口径不稳定（合约地址未公开、Dune 无专门仪表盘）；测试网数据无法预测主网商业化
- 处置：defer → 等 Arc 主网上线后，参照 `superpowers/plans/arc-mainnet-launch.md` 响应
- 学习：数据源未稳定前不要放入日监清单，浪费注意力且容易产生错误信号

**案例 N-03：管理层情绪评分**
- 假设：对财报电话会管理层语气（乐观/中性/谨慎）进行量化评分，作为前瞻信号
- 实验：设计一套5维度管理层情绪评分模板
- 评估：管理层语气评分主观性极强，且与分析师预期、Q&A 内容相互影响，难以标准化
- 处置：reject → 管理层表述只用"关键词摘录+直接引语"方式记录，不评分
- 学习：无法标准化的指标不要进入框架；定性信号用"引用"而非"评分"处理
