# 跨文件数据一致性验证矩阵

## 结论

本文件定义跨 metrics 文件的数据一致性检查规则，每周执行一次。
目标是发现同一指标在不同文件中填写不一致的情况，在研究结论层污染前拦截。

## 验证目的与执行频率

| 项目         | 内容                                                              |
| ------------ | ----------------------------------------------------------------- |
| 验证目的     | 确保日报、周报、财报、估值文件使用相同口径、相同数值、相同结论逻辑 |
| 执行频率     | 每周一次，在填写 `04-competition-dashboard.md` 之前完成           |
| 执行触发     | 正常：每周五复盘前；异常：任何文件更新后 24 小时内                |
| 执行记录位置 | 本文件末尾"检查记录"区域，按周追加                                |
| 责任方       | 完成当周周报的分析师                                              |

## 第一层：日报 → 周报一致性检查

`01-daily-watchlist.md` 与 `02-weekly-review.md` 的数据必须在以下项吻合。

| 检查编号 | 检查内容                                                                                                             | 允许误差 / 判断规则                                            | 不一致时动作                                                     |
| -------- | -------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------- | ---------------------------------------------------------------- |
| DW-01    | `01` 周四 USDC circulating supply 快照值，必须与 `04-competition-dashboard.md` 指标记录表的"USDC circulating supply 本周值"一致 | 允许偏差 ≤0.5%（来源口径差异）；>0.5% 标注并核查来源          | 优先使用 Circle Transparency 官方数据，更新非权威文件           |
| DW-02    | `01` 当周 net mint/redeem 累计方向（净铸造 / 净赎回），必须与 `02` 周报模板中"USDC 供应变化"方向一致                | 方向不允许矛盾（一个说净铸造，另一个说净赎回）                 | 重新核查 Circle Transparency，确认方向后同步两文件               |
| DW-03    | `01` 当周触发的 P0 事件（监管公告、储备报告异常、CRCL 放量跌破均线），必须在 `02` 周报的"竞争结构结论"或"下周重点"中得到反映 | 不允许 P0 事件在日报记录但周报无对应处理说明                   | 在 `02` 补充对应周的事件反映说明；若已过该周，在下周"遗留事项"注明 |
| DW-04    | `01` 当周 3M T-Bill 变化幅度（如单周变化 ≥25 bps），必须在 `02` 的周报模板中"转账量与活跃地址"或"结论"部分有利率背景注释 | 单周变化 <10 bps 可不注释；≥25 bps 必须注释                    | 在 `02` 对应周补充利率背景说明                                   |
| DW-05    | `01` 记录的 CRCL price/volume P0 信号（如放量跌破 20D/50D 均线），必须在 `02` 同周结论级别中体现（不能是"增强"结论） | P0 触发且当周结论为"增强"时，视为一致性错误                    | 重审 `02` 该周结论，说明 P0 信号是否已解除或被基本面覆盖         |

## 第二层：周报 → 财报一致性检查

`02-weekly-review.md` + `04-competition-dashboard.md` 与 `03-quarterly-earnings.md` 的数据必须在以下项吻合。

| 检查编号 | 检查内容                                                                                                                              | 允许误差 / 判断规则                                                 | 不一致时动作                                                      |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- | ----------------------------------------------------------------- |
| WE-01    | `04` 指标记录表的"RLDC margin（主，÷储备收入）本周值"，必须与 `03` 计算表中同一财季的 RLDC margin 自算值一致                        | 允许偏差 ≤0.1 ppt；>0.1 ppt 视为口径错误                            | 以 `03` SEC filing 自算值为准，更新 `04` 对应行；标注修改日期     |
| WE-02    | `04` Competition Score 的 D3 分销渠道议价档位，必须与 `03` 当季 RLDC margin 档位一致：margin <38% 时 D3 最高 25 分；≥42% 时 D3 最低 75 分 | 不允许 RLDC margin <38% 但 D3 打 50 分或以上                        | 重新计算当周 D3 并更新 competition score；触发异常处理记录        |
| WE-03    | `02` 周报中 competition score 的 D4 收益型稳定币蚕食档位，必须与 `03` 当季 Other revenue 趋势方向一致：D4 ≤25 分（蚕食加剧）时，`03` 的 Other revenue 不能出现"强化"结论 | D4 ≤25 分 + `03` Other revenue 强化 = 不一致                        | 检查 Other revenue 来源是否真正反映 USDC 使用质量，修正评分逻辑   |
| WE-04    | `02` 周报中 competition score 的 D7 平台化验证档位，必须与 `03` 当季 Other revenue share 档位一致：share <5% 时 D7 最高 25 分；share ≥10% 时 D7 最低 75 分 | 不允许 Other revenue share <5% 但 D7 打 50 分或以上                 | 重新计算 D7 并更新 competition score                              |
| WE-05    | `04` 指标记录表的"CPN TPV 本周值"（季度财报后唯一更新时机），必须与 `03` Autoresearch 十项财务复核中"Other revenue 可重复性"的判断方向一致 | CPN TPV 连续增长但"Other revenue 可重复性"为降级，视为矛盾          | 核查 CPN 收入确认口径；确认 TPV 增长是否已商业化                  |
| WE-06    | `02` 当周 competition score 总分档位（强化/中性偏强/中性偏弱/恶化），必须与 `03` 当季 RLDC margin 情景一致：RLDC margin <38% 时 competition score 不能处于"竞争结构强化（75-100）"区间 | margin <38% + competition score ≥75 = 不一致                        | 触发两文件结论层复核；以 `03` RLDC margin 客观数据为优先参照      |
| WE-07    | `04` 渠道议价记录表的"分销成本率"判断，必须与 `03` "distribution cost 弹性"复核判断方向一致（同时为改善或同时为恶化，不允许一个说改善另一个说恶化） | 方向矛盾（一个改善一个恶化）= 不一致                                 | 检查计算口径（有无季节性因素），对齐后更新两文件对应行            |

## 第三层：财报 → 估值一致性检查

`03-quarterly-earnings.md` 与 `valuation/` 目录文件的逻辑必须在以下项吻合。
本层检查在每季度财报 T+24h 后执行一次，非财报周跳过。

| 检查编号 | 检查内容                                                                                                                      | 允许误差 / 判断规则                                        | 不一致时动作                                                    |
| -------- | ----------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------- | --------------------------------------------------------------- |
| EV-01    | `03` 当季 RLDC margin 档位必须与 `valuation/01-scenario-model.md` 的情景选择一致：margin <38% 时，不允许处于 Bull case         | margin <38% + Bull case = 不一致                           | 触发情景降级检查；更新 `valuation/01` 情景选择并记录            |
| EV-02    | `03` Other revenue share ≥10%（平台化切换条件①满足），必须在 `valuation/00-valuation-framework.md` 中有对应的"条件①满足"记录  | share ≥10% 但估值文件未更新条件①记录 = 不一致              | T+24h 内更新估值文件条件①；不允许估值结论先于条件记录          |
| EV-03    | `03` Average USDC 实际值必须在 `valuation/01-scenario-model.md` 对应情景的 USDC 增长假设区间内；若超出区间，需触发情景修正   | 允许偏差 ±5%；>5% 需修正情景或在估值文件注明差异原因       | 更新情景模型 USDC 假设；不允许实际值超出情景假设但估值不变      |
| EV-04    | `03` Adjusted EBITDA margin 实际值若低于 `valuation/01-scenario-model.md` Bear case 假设，必须在 `risk/00-risk-map.md` 中触发信用层检查记录 | Adjusted EBITDA margin 低于 Bear case 假设即触发           | T+0 先更新 `risk/00`，T+24h 更新估值情景                        |
| EV-05    | `03` 十项财务复核中任何填写"降级"的复核项，必须在 `valuation/00-valuation-framework.md` 的目标价计算模板中有对应的折价或风险注释 | 降级项无估值折价注释 = 不一致                              | T+24h 内在估值文件添加对应注释                                  |

## missing_info 管理规则

### 触发条件

当任何文件中某指标格填写了 `missing_info` 时，进入以下追踪 SOP。

### 追踪 SOP

| 步骤 | 操作                                                                                      | 时限                  |
| ---- | ----------------------------------------------------------------------------------------- | --------------------- |
| 1    | 在本文件"missing_info 追踪表"新增一行，记录：文件名、单元格位置、缺失原因、预期可获得日期 | 发现后 24 小时内      |
| 2    | 判断该缺失是否影响当周 competition score；若影响，在 `04` 本周快照"missing_info"行注明    | 同步操作              |
| 3    | 每周检查追踪表中所有 `missing_info` 条目是否到达预期可获得日期                            | 每次执行本验证矩阵时  |
| 4    | 数据可获得时：填写实际值，从追踪表移除，在原文件添加"已补全 YYYY-MM-DD"标注               | 数据可获得后 24 小时内 |
| 5    | 超过预期可获得日期 7 天仍未获得：升级为"长期缺失"，修改预期日期，注明无法获取原因         | 到期后 24 小时内      |

### missing_info 追踪表

| 编号 | 文件                            | 单元格 / 指标           | 缺失原因           | 预期可获得日期 | 状态     | 补全日期 |
| ---- | ------------------------------- | ----------------------- | ------------------ | -------------- | -------- | -------- |
| M-001 | 04-competition-dashboard.md    | CPN TPV                 | Circle 未公开披露  | 下季度财报     | 待追踪   |          |
| M-002 | 04-competition-dashboard.md    | Coinbase products average USDC | 等 Coinbase 财报 | 季度财报后 48h | 待追踪   |          |

> 使用时按上表格式追加行，不删除历史记录。

## 发现不一致时的处理流程

### 判定规则

满足以下任一条件判定为"不一致"：

1. 同一指标数值在两个文件中偏差超出允许范围。
2. 同一指标的趋势方向（升/降/平）在两个文件中相反。
3. 一个文件有结论而对应文件未记录触发依据。

### 数据来源优先级（裁定准则）

| 优先级 | 来源类型                          |
| -----: | --------------------------------- |
|      1 | SEC filing / Circle 官方 Transparency 页面 |
|      2 | Circle earnings release / 10-Q 正文 |
|      3 | DefiLlama（与 CoinGecko 交叉验证） |
|      4 | Dune Analytics / TokenTerminal    |
|      5 | 第三方媒体或推断值（不可用于裁定） |

### 处理步骤

| 步骤 | 操作                                                   | 责任方          | 时限         |
| ---- | ------------------------------------------------------ | --------------- | ------------ |
| 1    | 记录不一致：编号、两文件各自数值、差异量               | 发现方          | 发现后 2 小时 |
| 2    | 按来源优先级确认权威数值                               | 发现方          | 当天内       |
| 3    | 更新非权威文件，在对应行添加"已修正 YYYY-MM-DD，原值 X" | 发现方          | 当天内       |
| 4    | 若不一致影响 competition score 或估值，重新计算并更新  | 发现方          | T+24h        |
| 5    | 在本周检查记录模板的"发现的不一致"列填写修正摘要        | 发现方          | 本周复盘前   |
| 6    | 同一检查项连续 3 周发现不一致，升级为"系统性口径问题"，修正来源 SOP | 任一方 | 第 3 周触发后 |

### 禁止事项

- 不允许取两文件数值的平均值作为最终值。
- 不允许在未裁定前将其中一个值标注为"参考"后继续使用另一个。
- 不允许因"差距很小"跳过步骤 3-5 的记录。
